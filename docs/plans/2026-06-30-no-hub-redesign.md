# ShareBox 架构重定位 — 去掉 Hub，回归 syncthing 原生模型

> 日期: 2026-06-30 | 状态: 设计确定，待实现
> 取代: `docs/plans/2026-06-18-family-fileshare-design.md`（三层 Hub/Agent 架构）

---

## 一、产品定位（先明确，决定一切取舍）

- **不做商业项目**，面向自用 + 亲朋好友。当前模式本身没有商业潜力，不为想象中的规模买单。
- **要有合理的安全性**，但按"家庭自用"量级，不堆企业级方案。
- **不想自建/依赖公网协调服务器**（这是放弃 Hub 的核心动机）。

由这个定位推出的核心决定：**不自研 Hub，把身份、发现、连接、加密全部交给 syncthing 原生能力。**

---

## 二、为什么去掉 Hub

原 Hub（`cmd/sharehub` + `internal/hub`）唯一干的实事是：帮设备**交换 syncthing device ID**（信令），让它们自动配对。它**不传文件、不碰文件、不是中继**。

但代价是：
- 一个要自己跑、自己运维、自己保证安全的公网服务。
- 全部安全焦虑都来自它（零鉴权、弱邀请码、无 HTTPS、元数据裸奔）。
- 单点：挂了就配不了对。

**结论**：Hub 提供的只是"省得手抄 device ID"这点便利，却带来全部安全负担。删掉它，用 syncthing 原生模型替代。

---

## 三、新架构：syncthing 原生 = "向日葵模型，但更安全"

| 能力 | 由谁提供 | 说明 |
|------|---------|------|
| 设备身份 | syncthing **Device ID** | 设备 TLS 证书公钥哈希，全球唯一，伪造不了。等价于"向日葵识别码" |
| 配对授权 | **双向添加** | A 要连 B，B 也必须把 A 加进来。光知道 ID 连不上你 = 天然安全 |
| 设备发现 | 公共 discovery（可自建 `stdiscosrv`） | 只存 "device ID → 最近 IP:端口"，零知识，不碰内容 |
| NAT 兜底 | 公共 relay（可自建 `strelaysrv`） | 只转发**端到端加密后**的密文，中继解不开 |
| 传输加密 | syncthing TLS / E2E | 文件内容全程加密，中间设施零知识 |
| "家庭组"效果 | syncthing **introducer** | 一台常开机当介绍人，自动把家人互相介绍 |
| 浏览不下载 | ShareBox **RemoteAccess 文件夹类型** | 已实现，与 Hub 无关 |
| 宿主留一份/本机不留 | ShareBox 清理闭环 + `ignoreDelete` | 已实现，与 Hub 无关 |

**对"不信任三方中转"的回应**：发现服务器只存地址映射、中继只转密文，两者都看不到任何文件内容。要彻底洁癖可自建 `stdiscosrv`/`strelaysrv`（本仓库已含），但家庭自用公共设施足够安全。

---

## 四、配对 UX：扫码 + 介绍人

- **A1 首次配对**：ShareBox 显示本机 Device ID 的二维码；对方扫码 → 双方各确认一次 → 连上。扫码只是替代手抄 52 位 ID，双向确认不能省（安全来源）。
- **A2 加第 N 台**：把家里常开机设为 introducer，新设备只跟它配对一次，介绍人自动把其他家人介绍给它。
  - 前提：家里需有一台**相对常开**的设备当介绍人（台式机/NAS）。
- **A3 删设备/撤销**：去中心，只能在**每台**自己设备上各删一次。家庭几台可接受。

## 五、方案②：用"家庭清单"文件夹替代 Hub 的注册表

这是去 Hub 后唯一需要新建的机制，用来满足"看到全家有哪些收件箱可投"这个加分项。**仍然零服务器**——用一个 syncthing 文件夹本身充当注册表。

### 机制

- 全家共享一个特殊文件夹："ShareBox 家庭清单"。
- 每台设备**只写自己的一个 JSON**（如 `<deviceID>.json`），内容：设备名、device ID、自己暴露的收件箱列表（folder id / label / type）。
- 每台只写自己那份 → **永不冲突**。
- syncthing 自动同步该文件夹 → 每台都有全家的自述 → ShareBox 读取并聚合，展示"全家有哪些收件箱"。
- 等价于把 Hub 的数据库变成自动同步的文件夹：同样的设备列表 + 目录注册表，零服务器。

### 用户逻辑（关键流程）

| 流程 | 用户看到的 | 背后做的 |
|------|-----------|---------|
| 配对 | 扫码加设备 | 自动把"家庭清单"文件夹共享给对方（第一个共享的文件夹）；introducer 传播 |
| 注册收件箱 | 新建收件箱、选目录 | (a) 把该收件箱共享给清单里的家庭设备；(b) 更新本机清单 JSON 加一条 |
| 看全家目录 | 收件箱总览列表 | 读清单文件夹里所有 JSON，聚合展示 |
| 投送 | 点"老爸的 RECEIVER" 拖文件 | 从清单取老爸 device ID（注册时已共享）→ 直接投送 |

### 需在实现时处理的边角

- **新人加入后看到老收件箱**：新人能看到清单（同步即得），但要能**投**，需老设备把已有收件箱补共享给新人。
  - 方案：靠 introducer 自动背书共享；或 ShareBox 检测清单里出现新设备时，自动把本机收件箱补共享给它。
  - 用户逻辑层面：加新人后"稍等几秒，老收件箱自动对他开放"。

## 六、明确接受的取舍（用户已确认）

| 取舍 | 说明 | 根因 |
|------|------|------|
| 撤销设备要每台各删 | 没有"一处删除全网生效" | 去中心，无中心权威 |
| 离线设备无法投送 | 对方不在线则文件挂起等待，上线才送达 | P2P 模型，无中心暂存（Hub 也救不了，它不传文件） |
| iOS 基本不可用 | 仅第三方付费 App 且后台同步受限 | syncthing 在 iOS 的系统级限制，与 Hub 无关 |

离线投送的部分缓解：家里常开机（介绍人/NAS）天然在线，发往它的收件箱不受影响；AI Agent 场景正是发往常开机，无离线问题。

## 七、安全模型（替代原 T2 的"给 Hub 加鉴权"）

- **没有自建公网服务 = 没有那一堆要自己做的鉴权活。**
- 准入 = syncthing 双向添加 + 确认（基于公钥身份，不存在撞库）。
- 传输 = syncthing TLS / E2E，中间设施零知识。
- 残余风险仅在"家庭清单文件夹"：它在全家可见（设计如此，等于原来 Hub 的"全家可见"），只含元数据（设备名、ID、收件箱 label），不含文件内容，且只在已配对的信任设备间同步。
- 洁癖增强（可选，非必须）：自建 `stdiscosrv` + `strelaysrv`，连地址映射和密文中继都不经过公共设施。

## 八、迁移：删什么、改什么

### 删除
- `cmd/sharehub/`（Hub 服务入口）
- `internal/hub/`（Hub 实现 + sqlite 依赖 `modernc.org/sqlite`，如无他用可从 go.mod 移除）
- 前端 `gui/sharebox/index.html` 中的家庭组面板那一整块（Hub API 调用：建组/加组/邀请码/`pairWithGroup`/`syncFoldersToHub` 等）
- 文档中 Hub 相关段落

### 改造（前端 `gui/sharebox/index.html`）
- 设备配对：新增本机 Device ID 二维码展示 + 扫码加设备（调 syncthing `/rest/config/devices`）。
- 全家目录发现：把原"调 Hub 拿成员/目录"换成"读家庭清单文件夹 + 写本机清单 JSON"。
- 注册收件箱：建文件夹 + 共享给家庭设备 + 更新清单 JSON（替代 `syncFoldersToHub`）。
- 投送通道：`openSharedFolder` 的对端 device ID 来源从 Hub 改为清单。

### 保留（与 Hub 无关，已实现）
- RemoteAccess 文件夹类型（后端全套）
- 上传 API、文件浏览/拖拽上传、目录选择器、新建文件夹
- 暂存清理闭环 + 宿主 `ignoreDelete`

## 九、实现状态（2026-06-30 代码完成，未编译/未跨设备实测）

均已实现，细节与勾选见 `docs/todo.md` T4。要点：

- 后端新增 `GET/POST /rest/sharebox/manifest`（`lib/api/api.go`）：读聚合全家清单 / 写本机 `<deviceID>.json`，经 folder Filesystem 读写并触发扫描。
- 前端（`gui/sharebox/index.html`）：本机身份 + 二维码（复用 `/qr/`）、添加设备（粘贴 ID + 介绍人勾选）、新建收件箱（receiveonly + `ignoreDelete` + 自动共享 + 写清单）、家庭清单聚合面板、点击对端收件箱建 RemoteAccess 投送通道、新设备补共享。

**两个实现期的关键决定（重要）**：

1. **`autoAcceptFolders=false`**：若 true，对端会全量下载我的收件箱，违背"不下载/不留副本"。因此改为：
   - 清单文件夹靠**双方各自创建同一固定 ID**（`sharebox-manifest`）+ 互相共享来同步，不依赖 auto-accept。
   - 投送通道在用户点击对端收件箱时**显式创建** RemoteAccess（同收件箱 ID）。
2. **收件箱保留改靠显式 `ignoreDelete=true`**（建收件箱时设置）。原 `model.go` 在 auto-accept 时设 `ignoreDelete` 的改动现在不在主流程触发，保留作防御，无害。

**待办**：跨设备实测（两台机器）；注意清单/暂存使用相对路径。
