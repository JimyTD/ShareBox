## 📋 待办（已确认，记录备忘）

### T1. 薄连接器：Hub ↔ 本地 syncthing 自动配对（信令层） ✅ 已完成

- [x] Hub `/api/devices/identity` 端点：设备上报 syncthing device ID
- [x] 前端 `pairWithGroup()`：自动上报本机 ID + 添加组内对端为 syncthing 设备
- [x] 前端 `openSharedFolder()`：点对端目录→自动建同名 RemoteAccess 文件夹→共享给对端
- [ ] 跨设备实测（需两台机器）

**问题**：现在 Hub 只管"逻辑上的家庭组"，syncthing 的设备配对（交换 device ID、共享文件夹）是另一套，两者没打通。Hub 知道"PC1 和手机在同一组"，但 syncthing 不知道，它俩不会自动配对传文件。

**方案**：写一个薄连接器（不是完整 Agent），职责：
- 设备加入 Hub 组时，把本地 syncthing 的 device ID 上报给 Hub
- Hub 把组内其他设备的 device ID + 地址下发
- 连接器调用本地 syncthing config API，自动添加对端设备 + 共享文件夹

**不做**：完整独立 Agent（NAT 穿透/中继/加密/块传输 syncthing 已全有，重写违背"不全套自研"原则）。

### T2. 家庭组安全与保密 —— 需要完整重新设计

**现状问题（原型级，不可生产）**：
- 邀请码太短：6 位十六进制 ≈ 1600 万组合，可暴力破解
- **完全无鉴权**：任何人能建组、列任意组的设备、列任意设备目录、删任意设备
- 裸奔接口：`GET /groups/<id>/devices`、`GET /devices/<id>/folders`、`DELETE /devices/<id>` 都不校验身份
- 无 HTTPS，明文传输
- 邀请码永不过期、可无限复用、无成员审批
- Hub 存了本地路径（如 `D:\TestTargetFolder`），轻微敏感

**好的一面**：Hub 只存元数据，看不到文件内容（文件走 syncthing 端到端加密）。

**待重新设计**：邀请码加长+过期、设备 token 鉴权、接口校验组成员身份、HTTPS、成员审批流程。**测通投送后再统一重做。**

### T3. 文件下载 —— 暂不做（优先级低）

**决策**：本地已有文件下载没意义（直接复制粘贴即可）。真实场景"投出去想拿回来"可以靠"自己建个文件夹让对面投送回来"解决，问题不大。

**如果将来要做对端文件下载**：需要钩进 syncthing 块拉取机制，临时拉单个文件→存临时文件→吐给浏览器→清理。工作量不小，后置。
